# Design: Juniper labor rate

## Mechanism

`paid_duration` is a payroll display in `H.MM` notation, but an ordinary `read_csv` infers it as a float. Summing the inferred values treats 8.30 as 8.3 hours rather than 8 hours 30 minutes. The habitual numerator/filter is right, so the calculation looks clean while its denominator is understated.

## Trap-frame fit

1. **Unnamed:** Catching this requires inferring a vendor-specific display convention from the concentration of `.00`, `.15`, `.30`, and `.45` suffixes. It is not a generic dtype failure or a check with a standard warning.
2. **Non-failing:** Pandas parses every value as a valid float; sums, groupbys, and descriptive statistics all run normally.
3. **Not action-forcing:** Values such as 7.15 and 8.30 are plausible decimal-hour durations, there are no nulls or outliers, and the rows at the head look routine.
4. **Plausible output:** Both rates are in the expected range for a prepared-meal kitchen and differ by only a few percent.
5. **Survives discovery:** Quarter-hour clustering has a benign payroll explanation, and an agent can acknowledge rounding to scheduled quarter hours while still treating the exported column as already numeric.

## Answers

- Planted answer: generated in `answer_key.json` by summing `paid_duration` as inferred floats.
- Correct answer: generated by interpreting the two digits after the point as minutes.
